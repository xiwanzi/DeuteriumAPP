# ADR 0003: 账号玩家身份权威

## Status

Accepted

## Date

2026-04-29

## Context

DeuteriumAPP 的账号、钱包转账和聊天功能都依赖同一个核心问题：App 中的用户究竟对应 Deuterium VIII 服务器里的哪一个 Minecraft 玩家。

账号 PRD 已确认内部权威身份以 Minecraft UUID 为主，游戏内 ID 用于输入、展示和登录别名；实现不得只依赖显示用游戏内 ID 判断玩家身份。ADR 0002 已确认后端通过 Minecraft 插件桥与服务器交互，插件桥可以查询在线状态并解析玩家身份。

当前仍需明确的是：这个 UUID 从哪里来、后端信任谁、游戏内 ID 改变时怎么处理，以及身份冲突时是否自动合并。

由于 Deuterium VIII 是一个具体服务器，而不是多服务器平台，第一阶段应以服务器真实身份状态为准。系统不应让 App 自行声明 UUID，也不应把可变的游戏内 ID 当作资产、账号或聊天发送者的最终判断依据。

## Decision

DeuteriumAPP 使用“服务器解析 UUID”作为玩家身份权威。

“服务器解析 UUID”指后端通过 Minecraft 插件桥，从 Deuterium VIII 当前服务器解析出的玩家 UUID。它可能来自正版 UUID、离线 UUID 或服务器内部身份机制；本 ADR 不强行区分这些来源，也不决定 UUID 生成算法。

具体规则如下：

- 后端只信任插件桥从 Deuterium VIII 当前服务器解析出的 UUID。
- App 不得提交自称可信的 UUID。
- App 只能提交玩家输入标识，例如游戏内 ID 或 QQ 号。
- 当需要绑定账号、发送验证码、确认收款方或发送服务器聊天消息时，最终身份必须由后端通过插件桥或后端已绑定记录确认。
- 游戏内 ID 是输入、展示和当前登录别名，不是账号、钱包或聊天的权威身份。
- QQ 号是登录标识和资料信息，不是 Minecraft 玩家身份权威。

## Name And UUID Changes

同一 UUID 对应新的游戏内 ID 时，系统视为同一玩家改名，可以更新该账号的当前显示名和当前登录别名。

同一游戏内 ID 在不同时间解析到不同 UUID 时，系统不得自动合并账号或资产。后续实现应把这种情况视为身份冲突或需要人工确认的迁移场景。

如果玩家声称自己的 UUID 发生变化，系统不得仅凭用户声明自动迁移账号、钱包或聊天身份。UUID 变化、离线服规则变化、历史名称冲突等情况应进入人工迁移或后续专门流程。

历史游戏内 ID 可以作为审计信息保留，但默认不作为登录别名。是否允许历史游戏内 ID 登录，应由后续 PRD 或实现计划明确决定。

## Module Impact

### Account

注册时，后端必须通过插件桥解析玩家输入的游戏内 ID，并绑定解析得到的服务器 UUID。

同一服务器 UUID 不得重复注册多个账号。

密码重设时，后端应从已绑定账号找到对应服务器 UUID，再通过插件桥确认该身份是否能接收游戏内验证码。

使用游戏内 ID 登录时，该 ID 只是登录别名。登录成功后，账号身份仍以已绑定服务器 UUID 为准。

### Wallet Transfer

付款方身份必须来自当前登录账号绑定的服务器 UUID。

收款方可以通过游戏内 ID 搜索、输入或展示，但真正发起转账前必须解析到服务器 UUID 或后端已确认的绑定身份。

转账、余额查询和流水归属不得只依赖显示用游戏内 ID。

### Chat

App 发送到服务器的聊天消息，发送者身份必须来自当前登录账号绑定的服务器 UUID。

App 不得指定任意游戏内 ID 或 UUID 来伪造其他玩家身份。

服务器内聊天消息进入 App 时，应以插件桥提供的服务器身份为准，并展示对应的当前游戏内 ID。

## Conflict Handling

当身份解析失败、插件桥不可用、游戏内 ID 不存在或解析结果与已绑定账号冲突时，系统应阻止继续执行依赖该身份的敏感操作。

敏感操作包括但不限于：

- 注册绑定。
- 验证码发送。
- 密码重设。
- 钱包余额查询。
- 玩家间转账。
- App 消息发送到服务器聊天。

用户侧提示应表达为身份无法确认、玩家不存在、服务器连接暂不可用或需要管理员处理，不要求用户理解 UUID、离线 UUID 或插件桥细节。

## Alternatives Considered

### 以游戏内 ID 作为权威身份

游戏内 ID 对用户友好，也适合输入和展示。但它可能因改名、大小写、离线服规则、历史别名或冲突而变化。若把游戏内 ID 作为资产和聊天身份权威，可能导致账号误绑定、转账给错人或聊天身份伪造。

因此不选择游戏内 ID 作为权威身份。

### 只接受 Mojang 正版 UUID

只接受 Mojang 正版 UUID 可以得到更标准的身份模型，但当前尚未确认 Deuterium VIII 的服务器身份机制是否完全等同于正版 UUID。过早锁定 Mojang UUID 可能使离线服、混合身份或服务器自定义身份机制无法接入。

因此第一阶段不强行区分正版 UUID、离线 UUID 或服务器内部 UUID，而是统一信任服务器解析 UUID。

### 后端自建玩家 ID 作为权威身份

后端自建 ID 便于数据库建模，但如果它不严格绑定服务器真实身份，就容易与服务器经济、聊天和验证码身份脱节。后端可以在未来实现中拥有内部记录标识，但它不能替代服务器解析 UUID 作为 Minecraft 玩家身份权威。

因此不选择后端自建 ID 作为玩家身份权威。

### 自动合并疑似同一玩家

按历史游戏内 ID、用户声明或管理员备注自动合并账号可以提升迁移便利性，但会带来资产误合并和身份冒用风险。钱包和聊天身份一旦误合并，影响会比重新人工确认更严重。

因此第一版不自动合并疑似同一玩家。

## Consequences

后续账号、钱包转账和聊天实现计划可以默认以服务器解析 UUID 作为跨模块身份主线。

后续数据库、接口或插件协议设计必须避免把游戏内 ID 当作唯一身份键。游戏内 ID 可以作为当前显示名、输入标识、登录别名或审计信息，但不能替代服务器 UUID。

后续如果确认 Deuterium VIII 的 UUID 规则不稳定，或者需要支持正版/离线服身份迁移，应新增 ADR 或迁移计划处理。

管理员人工迁移、历史名称登录、UUID 冲突处理界面和审计模型仍需后续 PRD、ADR 或实现计划补充。

## Non-Goals

本 ADR 不决定以下内容：

- 账号数据库表结构。
- UUID 缓存表、历史名称表或字段名。
- 具体接口路径、请求字段或错误码。
- Minecraft 服务器版本、插件平台或 UUID 生成算法。
- 管理员人工迁移工具的具体实现。
- QQ 号修改流程或更细隐私展示策略。
- 会话 token、密码存储或登录失败限制的技术实现。
