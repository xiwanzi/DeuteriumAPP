# ADR 0002: 后端 API 与 Minecraft 插件桥

## Status

Accepted

## Date

2026-04-29

## Context

DeuteriumAPP 第一阶段需要把 Android App、后端服务和 Deuterium VIII Minecraft 服务器连接起来，支撑账号验证码、钱包转账和聊天互通。

已有文档约束要求：

- Android App 连接独立后端 API。
- App 不直连数据库。
- App 不直接修改服务器经济数据。
- 后端通过 Minecraft 插件桥与服务器交互。
- Minecraft 插件不默认承担所有公网 API、账号安全和聊天长连接职责。

账号 PRD 已确认验证码由 App 请求触发，后端通过插件桥让服务器向在线玩家私聊发送验证码。后续钱包转账和聊天互通也都需要服务器侧能力参与：转账必须与服务器经济系统保持一致，聊天必须在 App 和服务器内聊天之间转发。

用户对插件桥的目标是“最简单，最稳定的连接方式”。在当前小型真实社区场景下，架构应优先减少 Minecraft 服务器公网暴露面，避免 App 或外部客户端直接接触服务器敏感能力，同时保持后端对账号、钱包和聊天请求的统一控制。

## Decision

DeuteriumAPP 第一阶段采用以下系统边界：

```text
Android App -> 后端 API -> Minecraft 插件桥 -> Deuterium VIII 服务器
```

插件桥采用 Minecraft 插件主动连接后端的 WebSocket 长连接。该连接是后端与 Minecraft 服务器之间的受控双向通道。

具体规则如下：

- Android App 只连接独立后端 API。
- 后端负责 App 鉴权、业务校验、限流、审计和请求编排。
- Minecraft 插件不直接面向 App，也不承担公网业务 API。
- Minecraft 插件启动后主动向后端建立 WebSocket 长连接。
- 后端通过该连接向插件下发受控服务器动作。
- 插件负责在 Minecraft 服务器侧执行受控能力，并把执行结果或服务器事件回传后端。
- 插件桥连接必须具备服务端鉴权、心跳、断线重连和明确的不可用状态。

第一阶段插件桥覆盖以下概念能力：

- 验证码投递：后端请求插件向在线玩家发送服务器私聊验证码。
- 玩家身份查询：后端请求插件确认玩家在线状态，并解析游戏内 ID 对应的可信玩家身份。
- 钱包转账执行：后端在完成 App 鉴权和业务校验后，请求插件调用服务器侧经济能力完成受控操作。
- 聊天互通：插件把服务器内聊天事件推送给后端，并接收后端下发的 App 聊天消息转发请求。

## Failure Strategy

插件桥离线、未鉴权、心跳超时或连接断开时，后端必须把插件桥视为不可用。

当插件桥不可用时：

- 注册验证码和密码重设验证码发送直接失败。
- 玩家在线状态和身份解析直接失败。
- 钱包转账等依赖服务器执行的请求直接失败。
- App 发送到服务器的聊天消息直接失败。
- 用户侧提示应表达为服务器连接暂不可用或稍后再试，不要求用户理解插件桥细节。

第一版不实现离线命令队列。尤其不得排队钱包转账请求，避免桥恢复后补执行导致资产一致性争议。

聊天消息第一版也不通过插件桥离线排队。是否支持聊天离线消息、历史消息或补发机制，应进入后续聊天 PRD、ADR 或实现计划。

## Security Boundaries

App 不得直接连接数据库、持有数据库凭据或直接修改服务器经济数据。

插件通信凭据、数据库凭据、服务器密钥、机器人 token 和其他敏感凭据不得提交到仓库。

插件桥只暴露给后端受控调用，不向 App 或未授权第三方开放服务器能力。

后端必须在下发插件动作前完成必要的鉴权和业务校验。插件不应信任来自 App 的原始身份声明，也不应让 App 指定任意服务器命令。

插件侧应只实现明确允许的能力，不提供通用远程命令执行入口。若未来确实需要新增服务器动作，必须通过 PRD、ADR 或实现计划明确用途、边界和失败处理。

## Alternatives Considered

### 后端直接调用插件暴露的公网接口

这种方案链路直观，但要求 Minecraft 服务器或插件暴露可被后端访问的公网入口。它会增加服务器防火墙、端口暴露、鉴权、防扫描和部署维护成本。

因此不选择该方案作为第一阶段默认架构。

### 插件轮询后端任务

插件轮询后端可以避免服务器暴露公网入口，也较容易理解。但它天然存在轮询间隔，聊天互通会有更明显延迟，验证码、转账等请求也会被建模成任务领取，后续处理超时、重复领取和状态确认会变复杂。

因此不选择轮询作为第一阶段默认插件桥协议。

### App 直接连接 Minecraft 插件或服务器数据库

该方案看似减少后端层，但会把服务器敏感能力和数据访问能力暴露给客户端，无法满足账号安全、钱包一致性、审计和限流要求。

因此明确禁止该方案。

### 后端与 Minecraft 服务器固定同机部署，通过本地 HTTP 或命令调用

如果部署形态固定在同一台机器，本地 HTTP 或本地命令调用可能更简单。但它会把架构绑定到单一部署方式，也不利于后续把后端和 Minecraft 服务器分开运维。

因此不把同机本地调用作为第一阶段架构决策。

## Consequences

后续账号、钱包转账和聊天实现计划可以默认依赖插件主动 WebSocket 长连接作为服务器侧通道。

后续后端实现必须维护插件桥在线状态，并在桥不可用时向 App 返回明确失败，而不是伪装成请求已受理。

后续钱包转账设计必须继续确认余额权威来源、经济插件能力和转账落账边界。本 ADR 只决定后端通过插件桥触达服务器侧能力，不决定余额存储或具体经济插件。

后续聊天设计仍需单独决定 App 与后端之间的实时通信方式。本 ADR 只决定后端与 Minecraft 服务器之间的插件桥通道。

后续如果需要多服务器接入、离线命令队列、复杂消息总线或插件桥消息 schema，应通过新的 ADR 或实现计划推进。

## Non-Goals

本 ADR 不决定以下内容：

- 后端语言、框架、部署方式。
- Minecraft 插件平台与服务器版本。
- 经济插件类型和余额权威来源。
- 具体 API 路径、请求字段、错误码或数据库 schema。
- WebSocket 具体消息格式、序列化格式、重试细节或超时时间。
- App 与后端之间的聊天实时协议。
- Android、后端或插件工程代码结构。
