# 信用点钱包与转账 PRD v1

## 1. 文档定位

本文档定义 DeuteriumAPP 第一版 `wallet-transfer` 模块的产品需求，覆盖信用点余额展示、余额刷新、收款人查找、玩家间转账、二次确认、基础流水和关键错误状态。

本文档遵循根目录 `AGENTS.md`、`CONTEXT.md`、`docs/prd/account.md` 以及已有 ADR 的约束：先文档后代码、服务 Deuterium VIII、App 不直连数据库、App 不直接修改服务器经济数据、后端通过插件桥与服务器交互、玩家身份以服务器解析 UUID 为权威。

本文档不定义数据库结构、具体 API 路径、请求字段、响应字段、错误码、XConomy API 调用方式、插件通信 schema 或 Android UI 实现。这些内容应在后续 ADR、实现计划或接口文档中确认。

当前版本日期：2026-04-29。

## 2. 目标

信用点钱包与转账的目标是让 Deuterium VIII 玩家在 Android App 中可信地查看自己的服务器信用点余额，并向其他服务器玩家发起快捷转账。

第一版需要做到：

- 玩家可以看到自己当前信用点余额。
- 玩家可以手动刷新余额，并理解余额刷新失败时的状态。
- 玩家可以通过游戏内 ID、QQ 号或玩家搜索入口找到收款玩家。
- 玩家可以向任意 Deuterium VIII 服务器玩家转账，不要求收款方已注册 App。
- 玩家在转账前必须确认收款玩家、金额和备注。
- 玩家可以看到基础信用点流水，用于确认最近收支和转账结果。
- 系统能明确反馈余额不足、收款玩家不存在、插件桥不可用、转账失败和结果未知等状态。

## 3. 用户与场景

### 3.1 主要用户

主要用户是已经注册并登录 DeuteriumAPP 的 Deuterium VIII 玩家。玩家已经通过账号体系绑定服务器解析 UUID，并希望在离开 Minecraft 客户端时仍能管理信用点。

### 3.2 核心场景

- 玩家不在电脑前，想确认自己还有多少信用点。
- 玩家需要快速给另一个服务器玩家转信用点。
- 玩家只知道对方游戏内 ID，需要搜索或输入后确认收款方。
- 玩家只知道对方 QQ 号，希望通过 QQ 查找已绑定 App 账号的玩家。
- 玩家转账后想确认交易是否成功。
- 玩家怀疑余额不对，需要查看最近流水。
- 服务器连接暂不可用时，玩家需要知道为什么无法刷新或转账。

## 4. 核心决策

### 4.1 模块边界

本模块使用 `wallet-transfer` 作为统一模块名，覆盖信用点余额、钱包视图、基础流水、玩家间转账、确认和失败处理。

钱包和转账不拆成两个独立 PRD，因为它们共享余额权威、玩家身份、交易结果和流水反馈。

### 4.2 信用点定义

信用点是 Deuterium VIII 服务器内已有的经济货币。当前服务器经济使用 XConomy。

App 中的钱包是服务器信用点资产的移动端视图，不是独立于服务器经济系统的新资产账户。

App 不得创造独立信用点余额，也不得在没有服务器确认的情况下把余额变更展示为已完成。

### 4.3 金额规则

- 信用点金额最多支持两位小数。
- 转账金额必须大于 0。
- 第一版不收手续费。
- 第一版不设置产品级单笔限额或每日限额。
- 如果服务器经济系统本身存在更严格的余额、精度或限制规则，最终结果以服务器侧确认为准。

### 4.4 余额刷新

- 玩家进入钱包页时，App 自动刷新一次信用点余额。
- 玩家可以在钱包页手动下拉刷新余额。
- 余额刷新失败时，App 应保留可理解的失败状态，不应把失败展示成余额为 0。
- 如果页面上仍有上一次成功刷新的余额，应明确区分当前展示是否为最近一次成功结果。

### 4.5 收款方查找

转账收款方可以通过以下入口查找：

- 输入游戏内 ID。
- 输入 QQ 号。
- 使用玩家搜索入口。

游戏内 ID 搜索面向 Deuterium VIII 服务器玩家。QQ 搜索只能命中已在 DeuteriumAPP 账号体系中绑定 QQ 的玩家。

无论从哪个入口查找，最终发起转账前都必须确认到一个服务器玩家身份。产品展示可以使用游戏内 ID、QQ 或其他用户可理解的信息，但身份判断必须遵守账号与身份 ADR：不得只凭显示名完成敏感操作。

### 4.6 转账确认

转账提交前必须二次确认。

确认内容至少从产品角度包含：

- 收款玩家。
- 转账金额。
- 可选备注。

玩家确认后才进入提交状态。提交过程中应锁定提交按钮，避免重复点击造成重复请求。

### 4.7 转账备注

转账备注为可选项。

玩家可以不填写备注直接转账。填写备注后，备注应在双方可见的相关流水中展示。

备注的长度、敏感词、可见范围和服务端保存策略不在本文档 v1 中细化，应在后续实现计划或聊天/内容安全相关文档中确认。

### 4.8 基础流水

第一版需要展示基础信用点流水，用于玩家查账和确认转账结果。

基础流水至少从产品角度展示：

- 收支方向。
- 对方玩家。
- 金额。
- 时间。
- 状态。
- 可选备注。

第一版流水不要求完整账单搜索、筛选、导出或复杂对账能力。

## 5. 功能需求

### 5.1 查看信用点余额

#### 用户目标

玩家打开 App 后可以看到自己当前的服务器信用点余额。

#### 前置条件

- 玩家已登录 App。
- 玩家账号已绑定服务器解析 UUID。
- 后端可以通过受控方式获取该玩家服务器信用点余额。

#### 基本流程

1. 玩家进入钱包页。
2. App 自动请求刷新信用点余额。
3. 系统确认当前登录玩家身份。
4. 系统获取该玩家服务器信用点余额。
5. App 展示余额和刷新状态。

#### 验收标准

- 已登录玩家可以看到自己的信用点余额。
- 余额展示不得依赖玩家手动输入自己的游戏内 ID。
- 余额归属必须基于当前登录账号绑定的服务器解析 UUID。
- 刷新失败时，不把失败结果展示为余额为 0。
- App 不直接连接数据库读取余额。
- App 不直接修改服务器经济数据。

### 5.2 手动刷新余额

#### 用户目标

玩家可以主动刷新余额，确认当前信用点状态。

#### 基本流程

1. 玩家在钱包页触发下拉刷新。
2. App 进入刷新中状态。
3. 系统重新获取玩家信用点余额。
4. 刷新成功时，App 更新余额展示。
5. 刷新失败时，App 展示失败提示，并允许玩家稍后重试。

#### 验收标准

- 玩家可以手动刷新余额。
- 刷新中状态不会造成页面重复跳动或误导。
- 刷新失败时，玩家能理解当前余额可能不是最新结果。

### 5.3 查找收款玩家

#### 用户目标

玩家可以找到要转账的收款方，并在提交前确认没有选错人。

#### 基本流程

1. 玩家进入转账入口。
2. 玩家输入游戏内 ID、QQ 号，或使用玩家搜索入口。
3. 系统返回可理解的候选玩家或明确的未找到状态。
4. 玩家选择或确认一个收款玩家。
5. App 在转账确认前展示收款玩家信息。

#### 验收标准

- 游戏内 ID 可以用于查找服务器玩家。
- QQ 号可以用于查找已绑定 QQ 的 App 玩家。
- QQ 未绑定任何玩家时，应提示未找到匹配玩家。
- 收款方不要求已注册 App，只要是可确认的 Deuterium VIII 服务器玩家即可。
- 转账前必须把收款方确认到服务器玩家身份。
- 不允许只凭用户输入文本直接完成转账。

### 5.4 发起信用点转账

#### 用户目标

玩家可以把自己的信用点转给另一个 Deuterium VIII 服务器玩家。

#### 前置条件

- 付款方已登录 App。
- 付款方账号已绑定服务器解析 UUID。
- 收款方已被确认到服务器玩家身份。
- 转账金额合法。

#### 基本流程

1. 玩家选择收款方。
2. 玩家输入转账金额。
3. 玩家可选填写备注。
4. App 展示二次确认内容。
5. 玩家确认转账。
6. App 锁定提交按钮并进入提交中状态。
7. 系统校验付款方身份、收款方身份、金额和余额。
8. 系统通过受控服务器侧能力完成信用点转账。
9. App 展示转账结果。
10. 转账成功后，钱包余额和基础流水应能反映该结果。

#### 验收标准

- 转账前必须二次确认。
- 转账金额最多两位小数。
- 转账金额必须大于 0。
- 付款方身份来自当前登录账号绑定的服务器解析 UUID。
- 收款方可以不是 App 注册用户，但必须是服务器可确认玩家。
- 提交中不能重复点击提交按钮。
- 第一版不收手续费。
- 第一版不设置产品级单笔或每日限额。
- App 不直接修改服务器经济数据。

### 5.5 查看基础流水

#### 用户目标

玩家可以查看最近信用点收支，确认转账是否成功，并回忆每笔转账用途。

#### 基本流程

1. 玩家进入钱包页或流水入口。
2. App 展示最近信用点流水。
3. 玩家可以看到收入、支出、对方玩家、金额、时间、状态和备注。
4. 当转账结果未知时，流水或结果页应帮助玩家理解需要稍后确认。

#### 验收标准

- 玩家可以看到基础信用点流水。
- 流水能区分收入和支出。
- 流水能展示转账状态。
- 有备注的转账可以展示备注。
- 流水不要求第一版支持筛选、搜索、导出或复杂对账。

## 6. 关键错误状态

### 6.1 余额刷新失败

触发场景：

- 网络不可用。
- 后端不可用。
- 插件桥不可用。
- 服务器侧余额读取失败。

产品要求：

- 不把失败结果展示为余额为 0。
- 提示余额刷新失败或服务器连接暂不可用。
- 允许玩家稍后重试。

### 6.2 插件桥不可用

触发场景：

- 后端无法通过插件桥获取余额、确认收款方或执行转账。

产品要求：

- 阻止继续完成依赖服务器侧能力的操作。
- 提示服务器连接暂不可用，请稍后再试。
- 不要求玩家理解插件桥细节。
- 不自动排队转账请求。

### 6.3 收款玩家不存在或身份无法确认

触发场景：

- 输入的游戏内 ID 无法识别。
- 搜索结果无法确认到服务器玩家身份。
- 身份解析失败或结果冲突。

产品要求：

- 阻止转账。
- 提示玩家检查收款方信息。
- 不允许只凭输入文本继续提交。

### 6.4 QQ 未绑定任何玩家

触发场景：

- 玩家输入 QQ 号查找收款方，但没有 App 账号绑定该 QQ。

产品要求：

- 提示未找到绑定该 QQ 的玩家。
- 可以引导玩家改用游戏内 ID 搜索。
- 不把 QQ 搜索失败解释为服务器玩家不存在。

### 6.5 金额不合法

触发场景：

- 金额为空。
- 金额小于或等于 0。
- 金额超过两位小数。
- 金额格式无法识别。

产品要求：

- 阻止进入确认或提交。
- 明确提示金额规则。

### 6.6 余额不足

触发场景：

- 付款方信用点余额不足以完成转账。

产品要求：

- 阻止转账成功。
- 明确提示余额不足。
- 不展示为系统错误。

### 6.7 转账提交中重复点击

触发场景：

- 玩家已经确认转账，提交尚未完成时继续点击提交。

产品要求：

- 提交按钮应锁定。
- 不发起多次转账。
- 页面应展示提交中状态。

### 6.8 转账失败

触发场景：

- 后端校验失败。
- 服务器侧执行失败。
- 插件桥返回失败。
- 余额或身份在提交时发生变化。

产品要求：

- 明确提示转账失败。
- 不自动重试。
- 提供返回修改或稍后再试的路径。

### 6.9 转账结果未知

触发场景：

- 请求中断。
- App 与后端连接中断。
- 后端无法向 App 确认最终结果。

产品要求：

- 不自动重试。
- 不提示玩家重复提交同一笔转账。
- 提示玩家稍后查看余额和流水确认结果。
- 后续实现计划必须继续处理结果未知下的幂等和查账体验，但本文档不定义具体技术方案。

## 7. 与账号和身份的关系

### 7.1 付款方身份

付款方必须来自当前登录账号绑定的服务器解析 UUID。

App 不得允许用户手动选择付款方，也不得允许用户伪造其他玩家身份发起转账。

### 7.2 收款方身份

收款方可以通过游戏内 ID、QQ 号或玩家搜索入口查找。最终执行转账前，收款方必须确认到 Deuterium VIII 服务器玩家身份。

如果收款方未注册 App，但服务器可以确认其玩家身份，第一版仍允许向该玩家转账。

### 7.3 QQ 查找边界

QQ 号只用于查找已绑定 QQ 的 App 玩家。未注册 App 或未绑定 QQ 的服务器玩家无法通过 QQ 被找到，但仍可通过游戏内 ID 或玩家搜索入口查找。

## 8. 非目标

信用点钱包与转账 PRD v1 不包含以下内容：

- 充值。
- 提现。
- 兑换。
- 手续费。
- 红包。
- 群转账。
- 管理员后台。
- 完整账单筛选、搜索、导出或对账工具。
- 离线转账队列。
- 自动重试转账。
- 钱包余额实时推送。
- 独立 App 钱包账户。
- XConomy API 调用方式。
- 数据库表结构。
- 具体 API 路径、请求字段、响应字段或错误码。
- 插件通信 schema。
- Android UI 技术实现。

## 9. 产品级概念操作

本文档只定义以下产品级概念操作：

- 查看信用点余额。
- 刷新信用点余额。
- 查找收款玩家。
- 发起信用点转账。
- 确认信用点转账。
- 查看基础信用点流水。

这些操作的接口路径、参数、错误码、存储模型、XConomy 调用方式、插件桥协议和 Android 实现由后续 ADR 或实现计划定义。

## 10. 验收清单

信用点钱包与转账 PRD v1 完成后，后续实现或计划必须能回答：

- 信用点是否被定义为 Deuterium VIII 服务器已有经济货币。
- App 是否没有创建独立钱包账户。
- App 是否不直连数据库。
- App 是否不直接修改服务器经济数据。
- 余额页是否进入时自动刷新。
- 玩家是否可以手动刷新余额。
- 刷新失败是否不会展示为余额 0。
- 金额是否最多支持两位小数。
- 转账金额是否必须大于 0。
- 第一版是否不收手续费。
- 第一版是否不设置产品级单笔或每日限额。
- 收款方是否可通过游戏内 ID、QQ 号或玩家搜索入口查找。
- QQ 搜索是否只命中已绑定 QQ 的 App 玩家。
- 收款方是否可以是不注册 App 的服务器玩家。
- 转账前是否必须二次确认。
- 提交中是否锁定提交按钮。
- 转账失败或结果未知时是否不自动重试。
- 基础流水是否能展示方向、对方玩家、金额、时间、状态和可选备注。
- 文档是否没有提前定义数据库表、具体接口、XConomy 调用方式或插件消息格式。

## 11. 待后续确认

以下问题不阻塞信用点钱包与转账 PRD v1，但需要在后续 ADR、计划或更细 PRD 中确认：

- 信用点余额权威读取方式。
- 转账由后端编排、插件调用 XConomy，还是由服务器侧经济能力完整执行。
- XConomy 的具体调用方式和失败返回语义。
- 转账幂等策略。
- 转账结果未知时的后端查账机制。
- 基础流水的数据来源和保留时长。
- 备注长度、敏感词和可见范围。
- 是否需要转账成功通知。
- 是否需要余额变化实时推送。
