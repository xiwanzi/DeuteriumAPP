# 账号体系 PRD v1

## 1. 文档定位

本文档定义 DeuteriumAPP 第一版账号体系的产品需求，覆盖注册、登录、验证码、QQ 号、密码重设、会话和关键错误状态。

本文档遵循根目录 `AGENTS.md` 与 `CONTEXT.md` 的约束：先文档后代码、服务 Deuterium VIII、原生 Android App、后端 API + Minecraft 插件桥、App 不直连数据库、不提前锁定 Android 技术栈。

本文档不定义数据库结构、具体 API 路径、请求字段、插件通信协议或 Android UI 技术栈。这些内容应在后续 ADR、实现计划或接口文档中确认。

当前版本日期：2026-04-29。

## 2. 目标

账号体系的目标是让 Deuterium VIII 的玩家用自己在 Minecraft 服务器内的身份安全地注册、登录和管理 DeuteriumAPP 账号，并为后续钱包、转账和聊天功能提供可信身份基础。

第一版账号体系需要做到：

- 玩家可以使用游戏内 ID、QQ 号、密码和游戏内验证码完成注册。
- 玩家可以使用“玩家 ID 或 QQ 号 + 密码”登录。
- 玩家可以通过游戏内验证码重设密码。
- App 能明确告诉玩家验证码、登录、注册和改密失败的原因。
- 后续钱包、转账、聊天功能可以依赖账号绑定后的 Minecraft UUID 作为权威身份。

## 3. 用户与场景

### 3.1 主要用户

主要用户是 Deuterium VIII 的 Minecraft 玩家。玩家已经在服务器内拥有游戏身份，并且可能拥有服务器经济余额、聊天身份和玩家间关系。

### 3.2 核心场景

- 新玩家第一次使用 App，需要绑定自己的服务器身份并注册账号。
- 已注册玩家打开 App，需要快速登录。
- 玩家忘记密码，需要证明自己仍然控制游戏内身份后重设密码。
- 玩家主动修改密码，需要通过游戏内验证码完成身份确认。
- 玩家退出当前设备登录状态。
- 后续钱包、转账和聊天功能需要使用已绑定的玩家身份。

## 4. 核心决策

### 4.1 身份权威

- 内部权威身份以 Minecraft UUID 为主。
- 游戏内 ID 用于玩家输入、用户可见展示和登录别名。
- 实现不得只依赖显示用游戏内 ID 判断玩家身份。
- 如果玩家改名、大小写变化或存在历史别名，系统仍应以 UUID 判断同一玩家。

游戏内 ID 与 UUID 的具体解析、缓存、更新和冲突处理不在本文档中定义，应进入后续账号架构 ADR 或实现计划。

### 4.2 验证码触发

- 验证码由 App 请求触发。
- 玩家在 App 中输入游戏内 ID 后，后端通过 Minecraft 插件桥请求服务器向该玩家发送私聊验证码。
- 验证码发送是产品行为，不锁定具体 Minecraft 命令实现。
- 玩家必须在线才能接收注册或改密验证码。

### 4.3 QQ 号规则

- QQ 号注册时必填。
- QQ 号全局唯一。
- QQ 号可作为登录标识。
- QQ 号在玩家资料页可见，默认指对已登录 App 用户可见。
- QQ 号的更细公开范围、隐藏开关、修改流程和隐私策略不在本文档 v1 中展开。

### 4.4 登录入口

- 登录页使用单一账号输入框。
- 单一账号输入框同时接受玩家 ID 或 QQ 号。
- 玩家无需在登录前选择“ID 登录”或“QQ 登录”。

### 4.5 密码重设

- 修改密码和忘记密码统一使用“游戏内验证码验证后重设密码”流程。
- 无论玩家是否已登录，只要要设置新密码，都必须通过游戏内验证码证明身份。
- 密码重设成功后，该账号既有登录会话失效。

## 5. 安全默认值

以下是账号 PRD v1 的产品默认值。后续实现可以在 ADR 或安全评审中收紧，但不得在没有说明的情况下弱化。

- 验证码格式：6 位数字。
- 验证码有效期：10 分钟。
- 验证码重发冷却：60 秒。
- 单个验证码最多尝试：5 次。
- 新验证码生成后，旧验证码立即失效。
- 密码最小长度：8。
- 密码最大长度：64。
- 连续登录失败限制：连续 5 次失败后，临时限制登录 15 分钟。
- 密码重设成功后：该账号既有登录会话全部失效。

密码强度的更细要求，例如大小写、数字、特殊字符、常见密码拦截，不在本文档 v1 强制定义。

## 6. 功能需求

### 6.1 注册

#### 用户目标

玩家用自己的 Minecraft 服务器身份创建 DeuteriumAPP 账号。

#### 前置条件

- 玩家拥有 Deuterium VIII 的游戏内 ID。
- 玩家当前在线，能接收服务器私聊消息。
- 玩家准备填写 QQ 号和密码。

#### 基本流程

1. 玩家打开注册页。
2. 玩家输入游戏内 ID、QQ 号和密码。
3. 玩家请求发送验证码。
4. 后端通过插件桥确认玩家在线，并解析该玩家对应的 Minecraft UUID。
5. 服务器向该在线玩家发送 6 位数字验证码私聊消息。
6. 玩家在 App 输入验证码。
7. 系统校验验证码、QQ 唯一性、密码规则和身份绑定状态。
8. 注册成功后，系统绑定 UUID、游戏内 ID、QQ 号和密码凭据。
9. 注册成功后，玩家可以进入 App 登录态。

#### 验收标准

- 玩家在线且输入信息有效时，可以收到验证码并完成注册。
- 注册完成后的账号内部绑定 Minecraft UUID。
- 同一个 UUID 不能重复注册多个账号。
- 同一个 QQ 号不能重复绑定多个账号。
- 注册成功后，玩家可以使用玩家 ID + 密码登录。
- 注册成功后，玩家可以使用 QQ 号 + 密码登录。
- 注册流程不会要求短信、邮箱、OAuth 或第三方登录。

### 6.2 登录

#### 用户目标

已注册玩家用最少步骤进入 App。

#### 基本流程

1. 玩家打开登录页。
2. 玩家在单一账号输入框中输入玩家 ID 或 QQ 号。
3. 玩家输入密码。
4. 系统判断账号标识类型并校验密码。
5. 登录成功后，当前设备进入登录态。

#### 验收标准

- 单一账号输入框支持玩家 ID 或 QQ 号。
- 使用玩家 ID + 密码可以登录。
- 使用 QQ 号 + 密码可以登录。
- 登录失败时，不向用户泄露过多账号枚举信息。
- 连续 5 次失败后，账号或当前登录尝试进入 15 分钟临时限制。

### 6.3 请求注册验证码

#### 用户目标

玩家在 App 中请求验证码，并在 Minecraft 服务器内收到私聊验证码。

#### 基本流程

1. 玩家输入游戏内 ID。
2. 玩家点击发送验证码。
3. 后端通过插件桥检查该游戏内 ID 对应玩家是否在线。
4. 玩家在线时，服务器向该玩家发送验证码。
5. App 提示验证码已发送。

#### 验收标准

- 验证码由 App 请求触发。
- 玩家不在线时，不生成可用验证码，并提示玩家先进入服务器。
- 同一玩家在 60 秒冷却期内不能重复请求验证码。
- 新验证码生成后，旧验证码失效。
- 验证码只通过服务器私聊发送，不在 App 响应中明文返回。

### 6.4 提交注册验证码

#### 用户目标

玩家把服务器私聊收到的验证码填回 App，完成身份证明。

#### 基本流程

1. 玩家收到 6 位数字验证码。
2. 玩家在 App 输入验证码。
3. 系统校验验证码是否存在、未过期、未超过尝试次数且匹配当前注册请求。
4. 校验成功后完成注册。

#### 验收标准

- 验证码错误时，提示验证码错误。
- 验证码过期时，提示重新获取验证码。
- 验证码尝试次数耗尽时，提示重新获取验证码。
- 验证码不能跨玩家、跨账号请求或跨用途复用。

### 6.5 密码重设

#### 用户目标

玩家在忘记密码或主动修改密码时，通过游戏内验证码确认身份并设置新密码。

#### 前置条件

- 玩家已注册。
- 玩家当前在线，能接收服务器私聊验证码。

#### 基本流程

1. 玩家进入密码重设入口。
2. 玩家输入玩家 ID 或 QQ 号。
3. 玩家请求发送重设验证码。
4. 后端识别账号绑定的 Minecraft UUID，并通过插件桥向在线玩家发送验证码。
5. 玩家在 App 输入验证码和新密码。
6. 系统校验验证码和新密码规则。
7. 系统更新密码。
8. 密码重设成功后，该账号既有登录会话失效。
9. 玩家使用新密码重新登录。

#### 验收标准

- 修改密码和忘记密码使用同一套验证码重设规则。
- 玩家不在线时，不能接收重设验证码。
- 重设验证码不能用于注册，也不能用于其他账号。
- 新密码必须满足长度规则。
- 密码重设成功后，旧密码无法继续登录。
- 密码重设成功后，既有会话失效。

### 6.6 退出登录

#### 用户目标

玩家可以从当前设备退出登录。

#### 基本流程

1. 玩家在 App 中触发退出登录。
2. 系统清除当前设备会话。
3. App 返回未登录状态。

#### 验收标准

- 退出登录后，当前设备不能继续访问需要登录的账号、钱包、转账或聊天能力。
- 退出登录只影响当前设备会话，不要求退出其他设备。

### 6.7 资料页 QQ 展示

#### 用户目标

已登录用户可以在玩家资料页看到玩家 QQ 号。

#### 验收标准

- QQ 号在资料页可见。
- QQ 可见范围默认为已登录 App 用户。
- 是否支持隐藏 QQ、修改 QQ 或仅好友可见，不在本文档 v1 范围内。

## 7. 关键错误状态

### 7.1 玩家不在线

触发场景：

- 注册请求验证码时，游戏内 ID 对应玩家不在线。
- 密码重设请求验证码时，账号绑定玩家不在线。

产品要求：

- 不生成可用验证码。
- 明确提示玩家需要先进入 Deuterium VIII 服务器。

### 7.2 游戏内 ID 不存在

触发场景：

- 输入的游戏内 ID 无法被服务器或插件桥识别。

产品要求：

- 提示玩家检查游戏内 ID。
- 不进入验证码发送成功状态。

### 7.3 QQ 号重复

触发场景：

- 注册时填写的 QQ 号已经绑定其他账号。

产品要求：

- 阻止注册。
- 提示 QQ 号已被使用。

### 7.4 UUID 已注册

触发场景：

- 同一 Minecraft UUID 已经绑定过 DeuteriumAPP 账号。

产品要求：

- 阻止重复注册。
- 引导玩家登录或走密码重设流程。

### 7.5 验证码错误

触发场景：

- 玩家输入的验证码与当前有效验证码不匹配。

产品要求：

- 提示验证码错误。
- 记录一次尝试次数。

### 7.6 验证码过期

触发场景：

- 验证码超过 10 分钟有效期。

产品要求：

- 阻止继续使用该验证码。
- 提示重新获取验证码。

### 7.7 验证码次数耗尽

触发场景：

- 单个验证码尝试达到 5 次。

产品要求：

- 该验证码失效。
- 提示重新获取验证码。

### 7.8 密码不合规

触发场景：

- 密码长度小于 8 或大于 64。

产品要求：

- 阻止提交。
- 明确提示密码长度要求。

### 7.9 账号或密码错误

触发场景：

- 登录标识不存在。
- 密码错误。

产品要求：

- 登录失败。
- 提示账号或密码错误。
- 不暴露过多账号存在性细节。

### 7.10 登录限制中

触发场景：

- 连续 5 次登录失败后，仍继续尝试登录。

产品要求：

- 阻止登录尝试。
- 提示 15 分钟后再试。

### 7.11 插件桥不可用

触发场景：

- 后端无法通过插件桥确认玩家在线、解析 UUID 或发送验证码。

产品要求：

- 不进入验证码发送成功状态。
- 提示服务器连接暂不可用，请稍后再试。
- 不要求玩家理解插件桥细节。

## 8. 会话规则

- 登录成功后，当前设备进入登录态。
- 退出登录清除当前设备会话。
- 密码重设成功后，该账号既有登录会话全部失效。
- 会话 token、刷新策略、过期时间和多设备管理不在本文档 v1 中定义，应进入后续安全 ADR 或实现计划。

## 9. 与其他功能的关系

### 9.1 钱包

钱包功能必须依赖账号绑定后的 Minecraft UUID 判断玩家身份。钱包不得仅凭显示用玩家 ID 查询或修改余额。

### 9.2 转账

转账功能必须使用已登录账号的 UUID 作为付款方身份来源。收款方可以通过玩家 ID 搜索或展示，但最终应解析到可信玩家身份。

### 9.3 聊天

聊天功能必须使用已登录账号绑定的 Minecraft 身份展示发送者。App 发送到服务器的消息不得伪造其他玩家身份。

## 10. 非目标

账号 PRD v1 不包含以下内容：

- 短信验证码。
- 邮箱注册、邮箱验证或邮箱找回。
- OAuth 或第三方登录。
- 离线发送验证码。
- 管理员后台。
- 数据库表结构。
- 具体 API 路径、请求字段或响应字段。
- Minecraft 插件通信协议。
- Android UI 技术栈选择。
- 钱包、转账、聊天的详细业务规则。
- QQ 隐藏开关、QQ 修改流程或更细隐私分级。

## 11. 产品级概念操作

本文档只定义以下产品级概念操作：

- 请求注册验证码。
- 提交注册。
- 登录。
- 请求密码重设验证码。
- 提交新密码。
- 退出登录。

这些操作的接口路径、参数、错误码、存储模型和插件桥协议由后续 ADR 或实现计划定义。

## 12. 验收清单

账号 PRD v1 完成后，后续实现或计划必须能回答：

- 注册是否绑定 Minecraft UUID，而不是只绑定显示用游戏内 ID。
- 验证码是否由 App 请求触发，并通过服务器私聊发送。
- 玩家不在线时是否无法获取注册或改密验证码。
- 验证码是否为 6 位数字。
- 验证码是否具备 10 分钟有效期、60 秒重发冷却、5 次尝试上限。
- 新验证码是否会让旧验证码失效。
- QQ 号是否注册时必填、全局唯一、可登录、资料页可见。
- 登录页是否使用单一账号输入框支持玩家 ID 或 QQ。
- 密码是否限制为 8-64 位。
- 连续 5 次登录失败后是否限制 15 分钟。
- 修改密码和忘记密码是否统一通过游戏内验证码重设。
- 密码重设成功后，既有会话是否失效。
- 文档是否没有提前定义数据库表、具体接口或 Android 技术栈。

## 13. 待后续确认

以下问题不阻塞账号 PRD v1，但需要在后续 ADR、计划或更细 PRD 中确认：

- 游戏内 ID 与 UUID 的解析来源。
- 服务器是否存在正版 UUID、离线 UUID 或自定义身份标识。
- 玩家改名后的展示名更新策略。
- QQ 号修改流程。
- QQ 号更细隐私展示策略。
- 会话有效期、刷新机制和多设备管理。
- 登录失败限制按账号、设备、IP 还是组合维度执行。
- 插件桥发送验证码的具体协议和失败重试策略。
