# Foundation 实施计划 v1

## 1. 文档定位

本文档定义 DeuteriumAPP 第一阶段基础工程与共享能力的实施计划，用于指导后续新窗口创建工程骨架、配置运行方式和建立跨模块约定。

本文档遵循根目录 `AGENTS.md`、`CONTEXT.md` 以及以下 ADR：

- `docs/adr/0001-android-ui-stack.md`
- `docs/adr/0002-backend-plugin-bridge.md`
- `docs/adr/0003-account-identity-authority.md`
- `docs/adr/0004-backend-plugin-tech-stack.md`

本文档不实现代码，不创建具体 API 契约，不设计完整数据库表结构，也不拆分 `docs/issues/`。

当前版本日期：2026-04-29。

## 2. 本次目标

Foundation 的目标是为 `account`、`wallet-transfer` 和 `chat` 三个业务模块提供统一的工程基础、运行边界和安全约束。

本计划覆盖：

- Android App、后端 API、Minecraft 插件三端工程骨架规划。
- Kotlin/JVM、Ktor、MySQL/MariaDB、Exposed、Flyway、Java 17 基线。
- Mohist 1.20.1 兼容插件路线：Spigot/Bukkit API、Java、Java-WebSocket。
- 后端 Windows 运行目录、一键启动脚本、配置模板和数据库迁移入口。
- 插件主动连接后端的 WebSocket 桥接基础。
- 账号、钱包转账和聊天共用的身份、配置、安全、日志和测试基线。

明确不做：

- 不创建 Android、后端或插件工程代码。
- 不定义具体 HTTP 路径、请求字段、响应字段或错误码。
- 不设计完整数据库 schema。
- 不实现业务逻辑。
- 不提交生产数据库密码、插件密钥、token 或本地生产配置。

## 3. 涉及端与职责

Android 端：

- 后续工程应采用 Kotlin + Jetpack Compose + Material 3。
- 第一阶段页面围绕账号、钱包转账和聊天组织，不引入 Web、iOS 或桌面端。
- Android 只连接后端 API，不连接数据库，不直接连接 Minecraft 插件。

后端：

- 后续工程应采用 Kotlin/JVM + Ktor。
- 后端负责 App 鉴权、业务校验、限流、审计、请求编排和插件桥 WebSocket 服务端。
- 后端连接 MySQL/MariaDB，通过 Exposed 访问数据，通过 Flyway 管理迁移。
- 后端运行时目标为 Java 17。

Minecraft 插件：

- 后续工程应以 Mohist 1.20.1 兼容为目标。
- 插件使用 Java、Spigot/Bukkit API 和 Java-WebSocket。
- 插件启动后主动连接后端 WebSocket 插件桥。
- 插件只实现明确允许的服务器能力，不提供通用远程命令执行入口。

数据库：

- 第一阶段数据库为生产环境已有 MySQL/MariaDB。
- 数据库用于账号、会话、审计、验证码、流水、聊天历史等后端记录。
- 具体表结构由各业务计划和后续实现推进，不在本文档完整定义。

文档：

- 本计划作为三个业务计划的基础约束。
- 若后续发现需要新增架构决策，应新增 ADR，而不是把长期技术决策散落在业务计划中。

## 4. 工程骨架计划

后续创建工程时，仓库应保持单仓库、多子工程或等价结构，至少能清晰区分：

- Android App 工程。
- Kotlin/Ktor 后端工程。
- Java/Bukkit 插件工程。
- 文档目录。

工程命名应使用英文标识，文档说明使用中文。

后续工程骨架必须能支持：

- 本地构建 Android App。
- 本地运行后端 API 和插件桥服务端。
- 构建可安装到 Mohist 1.20.1 的插件 jar。
- 生成可复制到生产 Windows 机器的后端运行目录。

工程骨架不应在第一步强行引入复杂模块化、微服务、多数据库、多端发布或 Docker Compose。

## 5. 后端运行目录计划

后端第一版交付物是可复制到生产 Windows 机器的运行目录。

运行目录应包含：

- 后端可执行包。
- Windows 一键启动脚本。
- 配置模板或配置说明。
- 数据库迁移入口。
- 日志目录或日志输出说明。
- 运行前检查说明。

生产配置必须通过本地配置文件或环境变量提供。配置模板可以提交到仓库，但真实生产配置不得提交。

配置至少需要覆盖以下类别：

- 后端监听端口。
- 数据库连接信息。
- 插件桥鉴权密钥或等价凭据。
- 会话、安全和限流相关配置。
- 日志级别。

具体配置文件格式由后续实现计划选择，但必须保证敏感值不会硬编码进代码或文档。

## 6. 插件桥基础计划

插件桥采用插件主动连接后端的 WebSocket 长连接。

Foundation 需要为后续业务模块预留以下基础能力：

- 插件连接鉴权。
- 心跳与在线状态判断。
- 断线重连。
- 请求与响应的关联。
- 插件桥不可用状态。
- 后端下发受控动作。
- 插件回传执行结果和服务器事件。

第一阶段插件桥概念能力包括：

- 向在线玩家投递验证码。
- 解析游戏内 ID 对应的服务器解析 UUID。
- 查询玩家在线状态和在线列表。
- 获取信用点余额。
- 执行信用点转账。
- 接收服务器公共聊天事件。
- 转发 App 公共聊天到服务器。

插件桥消息 schema、字段、序列化格式和超时细节不在本文档中定义，应在后续实现计划中按业务需要最小化确认。

## 7. 共享身份与安全基线

所有业务模块必须以服务器解析 UUID 作为 Minecraft 玩家身份权威。

共享规则如下：

- App 不得提交自称可信的 UUID。
- 游戏内 ID 只能作为输入、展示和当前登录别名。
- QQ 号只能作为登录标识和资料信息，不是 Minecraft 身份权威。
- 后端必须在敏感操作前确认当前登录账号与服务器解析 UUID 的关系。
- 插件不得信任 App 的原始身份声明。

敏感信息规则如下：

- 不提交数据库密码。
- 不提交插件桥密钥。
- 不提交 token、机器人凭据或生产配置。
- 不在日志中记录明文密码、验证码、会话 token 或完整敏感凭据。

安全能力应至少为后续模块预留：

- 密码哈希。
- 验证码有效期和尝试次数限制。
- 登录失败限制。
- 会话失效。
- 钱包和聊天操作审计。
- 插件桥鉴权和心跳。

## 8. 日志、测试与验证基线

后端日志应覆盖：

- 服务启动与配置加载结果。
- 数据库迁移结果。
- 插件桥连接、断开和鉴权失败。
- 账号、钱包转账和聊天的关键业务失败。

插件日志应覆盖：

- 插件启动与后端连接状态。
- WebSocket 重连。
- 受控动作执行成功或失败。
- 服务器事件监听异常。

Android 测试基线：

- 后续页面应优先覆盖关键表单校验、加载状态、错误状态和提交防重复。

后端测试基线：

- 后续后端应覆盖身份校验、验证码规则、登录限制、转账幂等和插件桥不可用。

插件测试基线：

- 后续插件应覆盖身份解析、玩家在线判断、XConomy 调用边界和聊天事件监听。

第一批文档阶段的验证重点是：计划能独立阅读，没有提前锁定未决策细节，没有违反已有 PRD 和 ADR。

## 9. 关键失败场景

Foundation 必须为以下失败场景提供统一处理方向：

- 后端无法连接数据库：服务启动失败或进入不可用状态，不能假装业务可用。
- 数据库迁移失败：阻止继续启动或明确进入维护状态。
- 插件桥未连接或心跳超时：依赖服务器能力的请求直接失败。
- 插件桥鉴权失败：拒绝连接并记录安全日志。
- 生产配置缺失：启动前明确报错，不使用危险默认密钥。
- App 网络不可用：Android 展示可理解的失败状态，允许用户稍后重试。

## 10. 验收标准

Foundation 计划完成后，后续实施必须能做到：

- 工程边界清楚区分 Android、后端、插件和文档。
- 后端技术栈符合 Kotlin/JVM + Ktor + MySQL/MariaDB + Exposed + Flyway。
- 插件技术栈符合 Mohist 1.20.1、Spigot/Bukkit API、Java、Java-WebSocket。
- Android UI 栈符合 Kotlin + Jetpack Compose + Material 3。
- 插件桥采用插件主动连接后端 WebSocket 长连接。
- 后端运行目录适配生产 Windows 机器和已有 MySQL/MariaDB。
- 真实密钥、密码、token 和生产配置不会提交到仓库。
- 三个业务模块共享服务器解析 UUID 作为身份权威。
- 文档没有创建代码工程、具体 API 字段或完整数据库表结构。
