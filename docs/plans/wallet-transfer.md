# Wallet Transfer 实施计划 v1

## 1. 文档定位

本文档定义 DeuteriumAPP 第一阶段 `wallet-transfer` 模块的端到端实施计划，用于指导后续新窗口实现信用点余额、收款方查找、玩家间转账、二次确认和基础流水。

本文档遵循：

- `AGENTS.md`
- `CONTEXT.md`
- `docs/prd/account.md`
- `docs/prd/wallet-transfer.md`
- `docs/adr/0002-backend-plugin-bridge.md`
- `docs/adr/0003-account-identity-authority.md`
- `docs/adr/0004-backend-plugin-tech-stack.md`
- `docs/plans/foundation.md`
- `docs/plans/account.md`

本文档不定义具体 API 路径、请求字段、响应字段、错误码、数据库表结构、XConomy 调用代码或 Android 页面代码。

当前版本日期：2026-04-29。

## 2. 本次目标

Wallet Transfer 的目标是让已登录玩家可信地查看自己的服务器信用点余额，并向 Deuterium VIII 服务器玩家发起快捷转账。

本计划覆盖：

- 查看信用点余额。
- 手动刷新余额。
- 通过游戏内 ID、QQ 号或玩家搜索入口查找收款玩家。
- 输入金额和可选备注。
- 转账前二次确认。
- 提交转账并防止重复提交。
- 展示基础信用点流水。
- 处理余额不足、收款方不存在、转账失败和结果未知。

明确不做：

- 不做充值、提现、兑换、手续费、红包、群转账。
- 不做管理员后台。
- 不做完整账单筛选、搜索、导出或复杂对账。
- 不做离线转账队列。
- 不做自动重试转账。
- 不让 App 直接修改服务器经济数据。
- 不创建具体数据库 schema 或接口字段。

## 3. 涉及端与职责

Android 端：

- 展示余额、刷新状态、收款方查找、金额输入、备注输入、二次确认和流水。
- 执行基础输入校验，例如金额大于 0、最多两位小数、备注非必填。
- 提交中锁定按钮，避免重复点击。
- 展示余额刷新失败、余额不足、转账失败、结果未知等状态。
- 只调用后端 API，不直连数据库或插件。

后端：

- 鉴权当前登录账号，并使用账号绑定服务器解析 UUID 作为付款方身份。
- 编排余额读取、收款方身份解析、转账执行和流水记录。
- 对金额、付款方、收款方、重复提交和插件桥状态做服务端校验。
- 请求插件调用服务器侧 XConomy 能力读取余额和执行转账。
- 保存基础流水和转账状态，用于查账和结果展示。

Minecraft 插件：

- 通过受控插件桥接收余额查询、玩家身份解析和转账执行请求。
- 调用服务器侧 XConomy 或等价经济能力。
- 返回余额、身份解析结果、转账成功或失败结果。
- 不暴露通用远程命令执行入口。

数据库：

- 保存转账记录、基础流水、请求状态、结果状态和必要审计信息。
- 具体表结构和字段名由后续实现确认。
- 不作为服务器信用点余额的最终权威来源。

文档：

- 若需要定义转账幂等键、结果未知查账策略或流水保留时长，应在后续实现计划中最小化明确。

## 4. 端到端数据流

余额展示数据流：

1. 玩家进入钱包页。
2. Android 请求后端刷新当前登录玩家余额。
3. 后端校验会话，并读取账号绑定服务器解析 UUID。
4. 后端通过插件桥请求插件读取该 UUID 对应信用点余额。
5. 插件调用 XConomy 或服务器经济能力并返回结果。
6. 后端返回余额和刷新状态。
7. Android 展示余额；失败时不把余额展示为 0。

收款方查找数据流：

1. 玩家输入游戏内 ID、QQ 号，或使用玩家搜索入口。
2. Android 请求后端查找候选收款方。
3. 后端根据入口分别处理：
   - 游戏内 ID 或玩家搜索：通过插件桥解析服务器玩家身份。
   - QQ 号：在账号体系中查找已绑定 QQ 的 App 玩家。
4. 后端返回可确认的候选玩家或未找到状态。
5. Android 展示候选玩家，玩家选择或确认收款方。

转账提交数据流：

1. 玩家选择收款方、输入金额和可选备注。
2. Android 展示二次确认内容。
3. 玩家确认后，Android 锁定提交按钮并请求后端提交。
4. 后端校验会话、付款方 UUID、收款方身份、金额格式和插件桥状态。
5. 后端记录转账请求进入处理中状态。
6. 后端通过插件桥请求插件调用 XConomy 执行转账。
7. 插件返回成功、失败或不可确认结果。
8. 后端更新转账状态和基础流水。
9. Android 展示成功、失败或结果未知。
10. 转账成功后，余额和流水应能反映结果。

基础流水数据流：

1. 玩家进入钱包页或流水入口。
2. Android 请求后端读取当前账号相关流水。
3. 后端基于当前登录账号 UUID 返回最近基础流水。
4. Android 展示方向、对方玩家、金额、时间、状态和备注。

## 5. 关键实现规则

金额规则：

- 金额必须大于 0。
- 金额最多两位小数。
- 第一版不收手续费。
- 第一版不设置产品级单笔或每日限额。
- 如果 XConomy 或服务器经济系统有更严格限制，以服务器确认结果为准。

身份规则：

- 付款方来自当前登录账号绑定的服务器解析 UUID。
- App 不允许用户手动选择付款方。
- 收款方可通过游戏内 ID、QQ 号或玩家搜索入口查找。
- 执行转账前，收款方必须确认到服务器玩家身份。
- 收款方可以不是 App 注册用户。

提交与幂等规则：

- Android 提交中必须锁定按钮。
- 后端必须防止同一提交被重复执行。
- 转账失败或结果未知时不自动重试。
- 结果未知时提示玩家稍后查看余额和流水确认。
- 后续实现应明确最小幂等策略，但不得通过离线队列补执行资产操作。

流水规则：

- 第一版展示基础信用点流水。
- 流水至少表达方向、对方玩家、金额、时间、状态和可选备注。
- 服务器内 `/pay` 成功交易应并入同一个基础流水接口，仍返回兼容 `WalletRecord`。
- 服务器内其他经济变动也应尽量并入同一个基础流水接口；能识别来源时写具体备注，不能识别时写为收入或支出并标记服务器经济变动。
- 备注可选；备注长度和敏感词策略待后续确认。

## 6. 关键失败场景

余额刷新失败：

- 网络、后端、插件桥或服务器经济能力失败时，Android 不把余额显示为 0。
- 如果有上一次成功余额，应明确这不是最新结果。

插件桥不可用：

- 余额读取、身份解析和转账执行失败。
- 不排队转账请求。
- Android 提示服务器连接暂不可用。

收款玩家不存在或身份无法确认：

- 阻止继续转账。
- Android 提示检查收款方信息。
- 不允许只凭输入文本提交。

QQ 未绑定任何玩家：

- 提示未找到绑定该 QQ 的玩家。
- 可引导玩家改用游戏内 ID 搜索。
- 不把 QQ 搜索失败解释成服务器玩家不存在。

金额不合法：

- 阻止进入确认或提交。
- Android 明确提示金额必须大于 0 且最多两位小数。

余额不足：

- 转账失败但不是系统错误。
- Android 明确提示余额不足。

提交中重复点击：

- Android 按钮锁定。
- 后端不重复执行同一笔转账。

转账失败：

- 后端或插件返回明确失败时，记录失败状态。
- Android 提示失败，并允许玩家返回修改或稍后再试。

转账结果未知：

- App 与后端连接中断或后端无法确认最终结果时，不提示重复提交。
- Android 提示稍后查看余额和流水。
- 后端应保留请求状态，后续通过流水或查询帮助确认。

## 7. 与账号和聊天的接口边界

Account 模块提供：

- 当前登录会话。
- 账号绑定服务器解析 UUID。
- 当前游戏内 ID 和 QQ 号。

Wallet Transfer 模块不负责：

- 注册和登录。
- 密码重设。
- 会话签发。
- 聊天消息同步。

Chat 模块可以从在线玩家侧边栏进入转账：

- 被点击的在线玩家作为收款方候选。
- Wallet Transfer 仍必须执行收款方确认和转账二次确认。
- 聊天侧边栏不能绕过金额校验、身份确认或提交防重复。

## 8. 验收标准

Wallet Transfer 实施完成后必须满足：

- 已登录玩家可以查看自己的信用点余额。
- 余额归属基于当前登录账号绑定服务器解析 UUID。
- 玩家进入钱包页时自动刷新余额。
- 玩家可以手动刷新余额。
- 刷新失败时不显示为余额 0。
- 转账金额必须大于 0，且最多两位小数。
- 第一版不收手续费，不设置产品级单笔或每日限额。
- 收款方可通过游戏内 ID、QQ 号或玩家搜索入口查找。
- QQ 搜索只命中已绑定 QQ 的 App 玩家。
- 收款方不要求注册 App，但必须是服务器可确认玩家。
- 转账前必须二次确认收款玩家、金额和备注。
- 提交中不能重复点击提交按钮。
- App 不直接修改服务器经济数据。
- 插件桥不可用时，不排队转账请求。
- 转账失败或结果未知时不自动重试。
- 基础流水展示方向、对方玩家、金额、时间、状态和可选备注。
- App 内流水时间按 UTC+8 展示。
- 用户开启钱包变动通知后，App 后台收到成功收入或支出流水时触发安卓通知。

## 9. 验证计划

文档阶段验证：

- 对照 `docs/prd/wallet-transfer.md` 检查余额、转账、流水和错误状态是否覆盖。
- 对照 ADR 0003 检查付款方和收款方都最终依赖服务器解析 UUID。
- 对照 ADR 0002 检查所有服务器经济能力都通过后端和插件桥。

后续实现阶段建议测试：

- 余额读取成功。
- 余额刷新失败保留可理解状态。
- 游戏内 ID 查找收款玩家成功。
- QQ 查找已绑定玩家成功。
- QQ 未绑定时提示未找到。
- 金额为空、为 0、负数或超过两位小数被阻止。
- 余额不足。
- 转账成功并更新流水。
- App 转账成功时，付款方显示支出流水；若收款方已注册 App，也显示收入流水。
- 服务器 `/pay` 和服务器经济余额变化写入流水后，后台 App 可收到钱包变动事件。
- 提交中重复点击不会重复转账。
- 插件桥不可用时转账失败且不排队。
- 转账结果未知时提示稍后查账。

## 10. 待确认假设

- XConomy 的具体 API 能力和失败返回语义。
- 插件侧执行转账时是否能天然保证原子性。
- 后端转账幂等键和超时策略。
- 结果未知后的查账机制。
- 基础流水的数据来源、保留时长和排序规则。
- 备注最大长度、敏感词和可见范围。
- 钱包变动通知采用一个总开关，默认关闭。
- 玩家搜索入口的匹配方式和候选数量。

## 11. v1.1 补充：钱包后台事件与通知

后端在成功写入 `wallet_records` 后，应通过 App WebSocket 向相关已注册 App 用户推送钱包变动事件。事件只用于刷新 App 钱包视图和后台系统通知，不改变旧的 `GET /wallet/records` 查询契约。

Android 钱包变动通知由用户在设置中手动开启，默认关闭。通知只在 App 后台触发；前台只更新余额和流水。流水显示时间固定使用 UTC+8，避免不同设备时区造成账务时间理解不一致。
