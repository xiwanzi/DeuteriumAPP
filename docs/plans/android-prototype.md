# Android 可交互原型实施计划 v1

## 1. 文档定位

本文档定义 DeuteriumAPP 第一版 Android 可交互原型的实施计划，用于指导当前阶段先把 App 体验跑起来，再通过真实交互继续澄清需求。

本文档遵循：

- `AGENTS.md`
- `CONTEXT.md`
- `docs/prd/account.md`
- `docs/prd/wallet-transfer.md`
- `docs/prd/chat.md`
- `docs/adr/0001-android-ui-stack.md`
- `docs/plans/foundation.md`

本文档只覆盖 Android 前端原型，不定义真实后端 API 契约、数据库 schema、Minecraft 插件协议或生产交付方式。

当前版本日期：2026-04-29。

## 2. 本次目标

Android 原型的目标是用本地假数据跑通账号、信用点钱包转账和聊天三条主流程，让用户能在手机体验中继续判断需求是否合理。

本计划覆盖：

- 创建 `android-app` Android 工程。
- 使用 Kotlin、Jetpack Compose 和 Material 3。
- 使用中文界面文案。
- 使用本地假数据模拟账号、钱包转账和聊天。
- 覆盖登录、注册、验证码、密码重设、钱包、转账、聊天、在线玩家和退出登录。

明确不做：

- 不接真实后端。
- 不实现数据库。
- 不实现 Minecraft 插件。
- 不提交生产密钥、数据库密码、插件桥密钥、token 或本地生产配置。
- 不把原型内部 repository 方法视为正式后端 API 契约。

## 3. 工程与原型边界

工程位置为 `android-app`。

Android 工程采用单 App 模块，包名为 `com.deuterium.app`，最小 Android 版本为 `minSdk 26`。

原型使用单 Activity 和 Compose 页面组织。第一版不引入复杂多模块结构，也不拆分后端或插件工程。

本地假数据层只承担原型交互：

- `AuthRepository`：登录、注册、验证码、密码重设、退出登录。
- `WalletRepository`：余额、刷新、收款方搜索、转账、流水。
- `ChatRepository`：最近消息、发送消息、在线玩家、服务器事件提示。

后续接真实后端时，可以保留页面与状态结构，再替换 repository 实现。

## 4. 页面与交互范围

未登录流程：

- 登录页：单一账号输入框，支持玩家 ID 或 QQ 号。
- 注册页：游戏内 ID、QQ 号、密码、请求验证码、提交验证码。
- 密码重设页：账号标识、请求验证码、新密码、提交验证码。

已登录流程：

- 钱包首页：信用点余额、刷新状态、最近流水、进入转账。
- 转账页：收款方搜索、金额、备注、二次确认、提交状态。
- 聊天页：最近公共聊天、发送消息、在线人数、在线玩家侧边栏、服务器事件提示开关。
- 个人页：当前玩家资料、QQ 号、退出登录。

聊天页在线玩家入口可以打开玩家基础资料，并进入转账流程。进入转账后仍必须经过金额校验和二次确认。

## 5. 校验与错误状态

原型必须覆盖以下本地校验：

- 密码长度为 8-64 位。
- 验证码为 6 位数字。
- 转账金额必须大于 0，且最多两位小数。
- 聊天消息不能为空、不能只包含空白字符，且最多 256 字符。

原型必须能演示以下错误状态：

- 玩家不在线。
- QQ 号重复。
- 验证码错误。
- 余额刷新失败。
- 收款方不存在。
- 余额不足。
- 服务器连接暂不可用。
- 聊天发送失败。

错误状态使用本地假数据或固定触发条件模拟，不代表正式后端错误码。

## 6. 验证计划

实现完成后优先验证：

- 注册成功后进入登录态。
- 登录成功后进入钱包页。
- 密码重设成功后回到登录流程。
- 余额刷新成功和失败状态可见。
- 转账必须二次确认。
- 转账成功后更新余额和流水。
- 余额不足和收款方不存在能阻止提交。
- 聊天可以发送本地消息。
- 服务器事件提示开关能改变聊天流展示。
- 在线玩家可以进入资料与转账入口。
- 退出登录后回到未登录状态。

如果本机未配置 Android SDK、ADB 或 Gradle，构建验证记录为环境阻塞，不得伪装成已完成。
