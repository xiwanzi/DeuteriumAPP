# CONTEXT.md

## 1. 文档定位

`CONTEXT.md` 记录 DeuteriumAPP 的项目背景、领域语言、长期约束、非目标和当前未决策事项。它是后续 `docs/prd/`、`docs/adr/`、`docs/plans/`、`docs/issues/` 的上下文来源。

本文件不直接定义完整产品需求，也不直接做架构决策。具体功能范围进入 `docs/prd/`，长期技术选择进入 `docs/adr/`，功能实施步骤进入 `docs/plans/`，任务拆分进入 `docs/issues/`。

当前上下文更新时间：2026-04-29。

## 2. 项目一句话

DeuteriumAPP 是面向 Minecraft 服务器 Deuterium VIII 玩家的一款原生 Android App，用来把服务器内的账号、钱包、转账和聊天体验延伸到手机端。

## 3. 项目背景

Deuterium VIII 是一个已有真实玩家活跃的 Minecraft 服务器，当前日活大约 20-40 人。项目不是从抽象市场机会出发，而是从一个具体服务器社区的实际需求出发：玩家已经在服务器中形成身份、经济关系和聊天关系，App 的价值在于让这些关系在离开游戏客户端后仍然可访问、可管理、可互动。

DeuteriumAPP 的第一阶段重点不是做一个通用 Minecraft 社区平台，也不是做一个复杂运营后台，而是服务 Deuterium VIII 的现有玩家，让他们可以更方便地处理服务器内的高频事务：

- 查看和管理自己的游戏内钱包。
- 与其他玩家进行快捷转账。
- 在 App 和服务器之间互通聊天。
- 用游戏内身份安全地注册、登录和找回密码。

因为服务器规模较小但真实活跃，产品应优先追求稳定、可信、轻量和易维护，而不是一开始追求复杂的平台化能力。

## 4. 目标用户

### 4.1 主要用户

主要用户是 Deuterium VIII 的服务器玩家。他们已经拥有 Minecraft 游戏内身份，并且会在服务器内参与经济活动、社交聊天和玩家间互动。

这些用户的典型诉求是：

- 不在电脑前时也能查看钱包余额。
- 可以快速给其他玩家转账。
- 可以在手机上继续参与服务器聊天。
- 可以用自己熟悉的游戏内身份完成 App 注册和登录。

### 4.2 项目维护者

项目维护者包括服务器管理者、未来开发者和 AI 代理。维护者需要清楚理解服务器语境、账号安全、钱包一致性和聊天互通边界，避免把小型服务器工具误设计成大型泛平台。

### 4.3 暂不服务的用户

第一阶段不以以下群体为主要用户：

- 非 Deuterium VIII 的 Minecraft 玩家。
- 多服务器联盟管理者。
- 公开社区平台访客。
- 复杂运营后台管理员。
- iOS、Web 或桌面端用户。

## 5. 产品目标

### 5.1 第一阶段目标

第一阶段目标是让 Deuterium VIII 玩家拥有一个可信的手机入口，完成账号、钱包、转账和聊天四类基础能力。

具体方向包括：

- 注册与登录：通过游戏内 ID、服务器验证码和密码建立 App 账号。
- 资料补充：注册后补充 QQ 号，QQ 号可作为登录标识。
- 密码修改：必须重新通过游戏内验证码验证身份。
- 钱包查看：同步并展示游戏内金钱数据。
- 钱包管理：在 App 内查看自己的钱包状态和必要的流水信息。
- 玩家转账：向其他服务器玩家发起快捷转账。
- 聊天互通：App 用户和服务器内玩家可以互相聊天。

### 5.2 长期目标倾向

长期目标可以逐步扩展，但必须围绕 Deuterium VIII 的真实社区需求。可能的长期方向包括：

- 更完善的钱包流水和转账记录。
- 更可靠的聊天消息同步与离线体验。
- 服务器公告、维护通知或活动提醒。
- 玩家资料页和基础服务器信息展示。
- 与服务器插件形成更稳定的双向通信协议。

这些长期方向只是上下文倾向，不代表已经进入第一阶段范围。

## 6. 设计原则

### 6.1 原生优先

DeuteriumAPP 是原生 Android App。界面、交互、性能和系统能力使用方式应符合 Android 用户的自然预期。

当前只确认“原生 Android”，尚未决定使用 Jetpack Compose 或 XML Views。任何代理不得在没有 ADR 的情况下擅自锁定 Android UI 技术栈。

### 6.2 服务器身份优先

App 账号必须围绕 Minecraft 玩家身份建立。玩家在服务器中的 ID 和身份绑定关系，是 App 账号可信度的核心来源。

第一阶段不默认引入短信、邮箱、OAuth 或复杂第三方账号体系。

### 6.3 钱包安全优先

钱包和转账直接影响玩家资产感知。任何余额展示、转账请求、失败处理和同步逻辑，都必须以安全和一致性优先。

App 不应直接连接数据库，也不应直接修改服务器经济数据。默认方向是 App 调用后端 API，后端通过插件桥与服务器交互。

### 6.4 聊天轻量优先

聊天的第一目标是服务器内外互通，而不是构建大型社交网络。除非后续 PRD 明确要求，第一阶段不默认包含复杂频道体系、好友系统、消息撤回、内容推荐或大型社区功能。

### 6.5 小规模真实社区优先

当前服务器日活约 20-40 人。产品和技术方案应适配这个量级：足够稳定、清晰、可维护，同时避免过早引入大型系统复杂度。

## 7. 领域语言

### 7.1 Deuterium VIII

Deuterium VIII 是当前 App 服务的 Minecraft 服务器名称。它不是泛指任意 Minecraft 服务器。

### 7.2 DeuteriumAPP

DeuteriumAPP 是当前 Android App 和仓库的暂定项目名。后续如果改名，应同步更新 `AGENTS.md`、`CONTEXT.md` 和相关 PRD/ADR。

### 7.3 玩家

玩家指 Deuterium VIII 的 Minecraft 玩家。默认情况下，玩家已经拥有游戏内身份，并可能拥有服务器经济余额和聊天身份。

### 7.4 游戏内 ID

游戏内 ID 指玩家在 Minecraft 服务器内可识别的名称。它用于注册、验证码投递和用户可见身份。

游戏内 ID 是否等同于 Minecraft 官方用户名、离线服用户名或服务器内部昵称，尚未确认。后续账号 PRD 和架构 ADR 必须明确它与 UUID、昵称、历史改名之间的关系。

### 7.5 Minecraft UUID

Minecraft UUID 是更稳定的玩家身份标识候选。当前尚未确认服务器是否使用正版 UUID、离线 UUID 或其他身份标识。账号绑定设计不得只依赖显示名，除非后续 ADR 明确接受该风险。

### 7.6 QQ 号

QQ 号是注册完成后的资料信息，同时可作为登录标识。QQ 号涉及个人信息，应避免收集无关数据，并在后续账号 PRD 中明确唯一性、修改流程和隐私展示策略。

### 7.7 验证码

验证码是 App 注册或密码修改时，用于证明玩家控制游戏内身份的一次性凭证。已确认的发送方式是通过服务器内私聊命令或等价机制，形如 `/msg 玩家id 验证码`。

验证码必须在后续设计中明确有效期、尝试次数、重发规则和过期处理。

### 7.8 钱包

钱包指玩家在 App 内看到的游戏内经济资产视图。钱包不是独立于服务器经济系统的新资产账户。

### 7.9 余额

余额指玩家在服务器经济系统中的金钱数值。余额的权威来源尚未确认，后续必须明确来自服务器插件、经济插件 API、数据库、缓存还是后端记录。

### 7.10 转账

转账指一个玩家向另一个服务器玩家转移游戏内金钱。转账必须由可信后端或服务器侧逻辑校验和执行，App 只负责发起请求和展示结果。

### 7.11 聊天互通

聊天互通指 App 中的聊天消息可以进入 Minecraft 服务器内聊天，服务器内聊天也可以被 App 用户看到。

### 7.12 后端 API

后端 API 指 Android App 直接调用的独立服务。它负责鉴权、业务校验、请求编排、安全控制和与插件桥通信。

### 7.13 插件桥

插件桥指后端与 Minecraft 服务器之间的通信层。它可以由 Minecraft 插件、服务器命令接口或其他受控机制实现，但不得把数据库凭据或服务器敏感能力暴露给 App。

## 8. 已确认约束

### 8.1 产品约束

- 第一阶段服务 Deuterium VIII，不做多服务器通用平台。
- 第一阶段产品形态为 Android App，不做 Web、iOS 或桌面端。
- App 必须遵循原生 Android 开发规范。
- 第一阶段功能围绕账号、钱包、转账和聊天互通展开。
- 文档默认使用中文，工程标识默认使用英文。

### 8.2 账号约束

- 注册使用游戏内 ID、验证码和密码。
- 验证码通过 Minecraft 服务器内私聊发送。
- 注册后补充 QQ 号。
- 登录支持玩家 ID + 密码，也支持 QQ 号 + 密码。
- 修改密码需要重新走游戏内验证码流程。

### 8.3 架构约束

- 默认采用“Android App -> 后端 API -> Minecraft 插件桥 -> 服务器”的架构方向。
- App 不直连数据库。
- App 不直接持有数据库凭据。
- App 不直接修改服务器经济数据。
- 技术栈选择必须进入 ADR 后再锁定。

### 8.4 协作约束

- 先文档后代码。
- 需求、架构、安全或数据流不清楚时，先补文档或继续访谈。
- `docs/plans/` 和 `docs/issues/` 默认按功能组组织。
- 当前阶段不生成 Android 工程骨架，不设计数据库表，不写具体 API 契约。

## 9. 非目标

第一阶段明确不做或不默认做以下事项：

- 不做通用 Minecraft 社区平台。
- 不做多服务器接入和跨服身份系统。
- 不做 Web App、iOS App、桌面 App 或小程序。
- 不做复杂运营后台。
- 不做大型社交网络能力，例如好友推荐、广场、内容流。
- 不默认支持第三方 OAuth、短信验证码或邮箱找回。
- 不默认实现复杂频道、撤回、私聊、群聊或消息审核后台。
- 不直接让 App 连接数据库。
- 不在技术栈未决策前创建 Android 工程骨架。

这些非目标可以在未来被重新讨论，但必须通过 PRD 或 ADR 明确改变。

## 10. 关键风险

### 10.1 身份绑定风险

如果游戏内 ID 不是稳定身份，玩家改名、重名或离线服 UUID 变化可能导致账号绑定错误。后续账号设计必须确认服务器的身份机制。

### 10.2 钱包一致性风险

钱包余额和转账必须与服务器经济系统一致。如果 App、后端和服务器之间存在重复写入、延迟同步或失败重试不当，可能造成玩家资产争议。

### 10.3 聊天实时性风险

聊天互通涉及实时消息、断线重连、重复消息和身份显示。第一阶段需要控制范围，先保证基本互通可靠，再考虑复杂聊天体验。

### 10.4 安全与滥用风险

注册验证码、登录密码、QQ 号、钱包转账和聊天发送都可能被滥用。后续设计需要考虑限流、会话过期、失败次数限制、敏感操作确认和服务端审计。

### 10.5 维护复杂度风险

项目服务一个小规模真实社区，过早引入复杂微服务、多端、多数据库或过重前端架构，会让维护成本高于收益。

## 11. 当前未决策事项

以下事项暂时只记录为待确认，不作为实现依据：

- Android UI 技术栈：Jetpack Compose 或 XML Views。
- Android 最低版本、目标 SDK 和发布渠道。
- Android 工程结构、模块划分、导航和状态管理。
- 后端语言、框架、部署环境和运行方式。
- Minecraft 服务器版本、插件平台和当前经济插件。
- 玩家身份权威标识：游戏内 ID、UUID 或服务器内部账号。
- 钱包余额权威来源和同步方式。
- 转账由后端执行、插件执行还是服务器经济插件执行。
- 聊天互通协议：轮询、WebSocket、长连接或插件推送。
- 是否需要聊天历史、离线消息、频道、私聊或敏感词。
- QQ 号的唯一性、可见性、修改流程和隐私边界。
- 管理员是否需要单独后台能力。

建议继续按以下顺序推进：

2. `docs/adr/android-ui-stack.md`：Android UI 技术栈选择。
3. `docs/adr/backend-plugin-bridge.md`：后端 API 与 Minecraft 插件桥架构。
4. `docs/prd/wallet-transfer.md`：钱包、余额同步、转账和流水。

如果用户希望继续“一份一份来”，下一份优先建议写 `docs/prd/account.md`，因为账号体系会影响钱包、转账和聊天的所有身份边界。
