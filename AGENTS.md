# AGENTS.md

## 1. 文档定位

`AGENTS.md` 是 DeuteriumAPP 项目的 AI 协作总规范。所有参与本项目的 AI 代理，包括 Codex、Claude 以及其他代码或文档代理，都必须先阅读并遵守本文件，再进行需求整理、文档编写、代码实现、测试验证或提交变更。

本文件优先服务 AI 代理，不替代面向人类贡献者的 README、PRD 或开发手册。若本文件与后续更具体的 `docs/adr/`、`docs/prd/`、`docs/plans/` 文档冲突，优先级如下：

1. 用户在当前对话中的最新明确指令。
2. 已接受的 ADR。
3. 已确认的 PRD 或计划文档。
4. 本 `AGENTS.md`。
5. 代理自己的默认习惯。

## 2. 项目事实

- 项目暂定名：`DeuteriumAPP`。
- 服务对象：Minecraft 服务器 `Deuterium VIII` 的玩家。
- 当前服务器规模：日活约 20-40 人。
- 产品形态：原生 Android App。
- 主要用户：服务器玩家。
- 第一阶段产品方向：
  - 账号注册、登录、身份绑定与密码修改。
  - 同步游戏内金钱数据。
  - 在 App 内管理自己的钱包。
  - 快捷转账给其他服务器玩家。
  - App 聊天与 Minecraft 服务器内聊天互通。

这些事实是当前项目的基础上下文。任何代理不得在没有用户确认或文档决策的情况下，把项目扩展成通用社区平台、跨服平台、Web 优先产品、桌面产品或非原生 Android 产品。

## 3. 核心协作原则

### 3.1 先文档后代码

当需求、边界、架构、安全策略、数据流或验收标准不清晰时，代理必须先补充或更新文档计划，而不是直接写代码。

允许直接实现的情况：

- 变更范围很小，且已有文档或用户指令已经明确目标、接口、验收标准。
- 修正文档中的错别字、格式问题或明显不一致。
- 执行用户明确要求的单一步骤，且不会引入架构或产品决策。

必须先补文档或询问用户的情况：

- 涉及账号、密码、验证码、权限、钱包余额、转账、聊天记录等敏感流程。
- 涉及 Android 技术栈选择、后端接口形态、插件桥协议、数据库模型。
- 涉及公网访问、鉴权、限流、消息实时性、数据一致性。
- 涉及新增功能模块、跨模块重构或长期维护约定。

### 3.2 小步推进

本项目从空仓库启动，代理应按文档体系逐步建立项目颗粒度。不要一次性生成大量未经确认的业务细节、数据库表、API 契约或 Android 工程代码。

每次工作都应尽量做到：

- 明确本次只处理哪一份文档或哪一个功能组。
- 明确哪些信息来自用户确认，哪些是临时假设。
- 保持变更可审阅、可回滚、可继续迭代。

### 3.3 中文文档，英文标识

- 项目文档默认使用中文。
- 代码包名、类名、函数名、API 字段、数据库字段、分支名和提交信息默认使用英文。
- 专有名词首次出现时应尽量解释清楚，避免只写缩写。

## 4. 文档体系约定

后续文档按以下职责拆分。当前第一版只创建 `AGENTS.md`，不要因为看到这些路径就自动创建空目录或空文件。

- `CONTEXT.md`：记录项目背景、服务器语境、领域语言、长期约束和非目标。
- `docs/prd/`：记录产品需求，说明用户、场景、功能边界、验收标准。
- `docs/adr/`：记录架构决策，包括 Android 技术栈、后端架构、插件桥、数据存储、安全策略等。
- `docs/plans/`：记录功能实现计划，按功能组拆分。
- `docs/issues/`：记录功能级问题拆分，按功能组组织。

### 4.1 文档颗粒度

- `CONTEXT.md` 定位为“项目总览 + 领域词典”，用于记录项目背景、领域术语、模块边界、关系、歧义和长期约束。
- PRD 写到产品规则、用户流程、错误状态和验收标准为止，不写接口字段、数据库表结构或具体技术方案。
- ADR 每个核心模块至少一份模块级架构决策，记录边界、权威来源、通信方式和安全取舍；ADR 不替代 PRD。
- `docs/plans/` 以一个功能纵切为单位，覆盖端到端实施范围、涉及端、数据流、失败场景和验收方式。
- `docs/issues/` 以“用户故事 + 验收切片”组织，不写成零散技术 TODO。

### 4.2 核心模块

`docs/plans/` 和 `docs/issues/` 的默认颗粒度围绕核心模块展开。当前核心模块统一为：

- `account`：注册、登录、身份绑定、密码修改、QQ 信息。
- `wallet-transfer`：余额同步、钱包展示、流水、玩家间转账、确认和失败处理。
- `chat`：App 与服务器互通聊天、消息同步、在线状态。

钱包和转账默认合并为 `wallet-transfer`，因为它们共享余额权威、一致性、流水和交易边界。代理不得擅自把 `wallet-transfer` 拆回两个模块，除非后续 PRD、ADR 或计划文档明确要求。

代理不得把一个大功能拆成过细的零散任务，除非已有计划文档要求这样做。

## 5. 产品与领域约束

### 5.1 账号体系

默认账号流程如下：

- 玩家使用游戏内 ID、验证码和密码注册。
- 验证码通过 Minecraft 服务器内私聊形式发送，形如：`/msg 玩家id 验证码`。
- 注册完成后，玩家补充 QQ 号作为资料信息。
- 登录方式支持：
  - 玩家 ID + 密码。
  - QQ 号 + 密码。
- 修改密码必须重新走游戏内验证码验证流程。

涉及账号的设计必须优先考虑：

- Minecraft 玩家身份与 App 账号的绑定关系。
- 验证码有效期、次数限制和重放风险。
- 密码存储、登录失败限制和会话过期。
- QQ 号作为登录标识时的唯一性与修改流程。

在没有 ADR 或 PRD 明确前，不要默认引入第三方 OAuth、短信验证码、邮箱验证或复杂账号找回体系。

### 5.2 钱包与转账

钱包功能必须与服务器内经济系统保持一致。代理在设计或实现前必须明确：

- 游戏内金钱数据的权威来源。
- App 展示余额的同步时机。
- 转账请求由谁校验、谁落账、谁负责回滚。
- 转账成功、失败、重复提交和网络中断时的用户反馈。

除非已有 ADR 明确允许，App 不得直接修改游戏经济数据，也不得直接连接数据库执行余额变更。

### 5.3 聊天互通

聊天功能目标是让 App 玩家和服务器内玩家互相聊天。代理在设计或实现前必须明确：

- App 用户与 Minecraft 玩家身份如何映射。
- 服务器内消息如何进入后端。
- App 消息如何转发到服务器。
- 是否需要历史消息、离线消息、撤回、敏感词、权限或频道。

第一阶段不要默认扩展成大型社交系统，除非用户或 PRD 明确要求。

## 6. 工程与架构边界

### 6.1 Android 端

Android 端必须遵循原生 Android 开发规范。当前只锁定“原生 Android”，暂不锁定 Kotlin + Compose 或 Kotlin + XML Views。

代理不得在没有 ADR 的情况下擅自决定：

- 使用 Jetpack Compose 还是 XML Views。
- 模块化方案。
- 状态管理方案。
- 网络库、依赖注入库、本地数据库或导航方案。
- 最低 Android 版本和目标 SDK。

在 Android 技术栈尚未决策前，相关讨论应进入 `docs/adr/`，不要直接生成 Android 工程骨架。

### 6.2 后端与插件桥

默认架构方向是：

1. Android App 连接独立后端 API。
2. 后端负责鉴权、业务校验、钱包与聊天请求编排。
3. 后端通过 Minecraft 插件桥与 `Deuterium VIII` 服务器交互。

禁止默认采用以下方案：

- App 直连数据库。
- App 直接持有数据库凭据。
- App 直接修改服务器经济数据。
- Minecraft 插件独自承担所有公网 API、账号安全和聊天长连接职责。

如果确实需要偏离“后端 API + 插件桥”，必须先写 ADR 并说明原因、风险和替代方案。

### 6.3 安全底线

涉及以下内容时，代理必须显式考虑安全：

- 密码与验证码。
- 玩家身份绑定。
- QQ 号登录。
- 钱包余额与转账。
- 聊天消息发送。
- 后端与 Minecraft 插件桥通信。

不得把明文密码、数据库密码、服务器密钥、机器人 token、插件通信密钥提交到仓库。

## 7. AI 工作流程

### 7.1 接到需求时

代理应先判断需求属于哪一类：

- 需求澄清：优先访谈用户，沉淀到 `CONTEXT.md` 或 `docs/prd/`。
- 架构决策：优先产出或更新 `docs/adr/`。
- 功能实现：先确认是否已有 PRD、ADR 和 plan。
- 问题拆分：写入 `docs/issues/`，并保持功能组颗粒度。
- 代码实现：只在文档足够明确时进行。

### 7.2 编写计划时

计划必须说明：

- 本次目标和不做什么。
- 涉及哪些端：Android、后端、Minecraft 插件、文档。
- 数据如何流动。
- 关键失败场景。
- 验收标准。
- 需要用户确认的假设。

### 7.3 修改代码时

代理应遵守：

- 先阅读现有文件和相邻约定，再修改。
- 优先沿用项目已有技术栈和风格。
- 不做无关重构。
- 不回滚用户或其他代理的未相关改动。
- 不在需求未确认时新增大规模框架代码。
- 对账号、钱包、转账、聊天相关变更保持更高测试要求。

### 7.4 完成工作时

代理最终回复应说明：

- 改了什么。
- 如何验证。
- 哪些内容仍是假设或待确认。
- 后续最自然的一步是什么。

## 8. 测试与验证要求

当前仓库尚未建立代码工程。文档阶段的验证重点是：

- 文档能否独立阅读。
- 文档是否覆盖已确认的项目事实。
- 文档是否避免提前锁定未决策的技术栈。
- 文档是否明确禁止高风险默认方案，例如 App 直连数据库。

未来进入代码阶段后，代理应根据模块风险选择测试：

- 账号模块：注册、登录、验证码、改密、登录失败限制。
- 钱包模块：余额同步、展示一致性、接口失败。
- 转账模块：成功转账、余额不足、重复提交、并发请求、网络中断。
- 聊天模块：消息发送、接收、重连、重复消息、身份显示。
- 插件桥：服务器在线/离线、命令执行失败、超时、鉴权失败。

## 9. Git 与发布约定

当前仓库目标地址：`https://github.com/xiwanzi/DeuteriumAPP`。

初始文档版本完成后，默认直接推送到远程 `main` 分支。后续进入代码阶段后，除非用户另有明确指令，较大变更应使用独立分支和 Pull Request。

提交前应确认：

- 本次变更范围清晰。
- 没有混入无关文件。
- 没有提交密钥、凭据或本地临时文件。
- 文档内容与用户最新确认的信息一致。

## 10. 当前未决策事项

以下内容尚未决策，代理不得擅自锁定：

- Android UI 技术栈：Compose 或 XML Views。
- Android 工程结构和模块划分。
- 后端语言、框架和部署方式。
- Minecraft 插件平台与版本。
- 数据库类型和表结构。
- 钱包数据的权威来源和同步协议。
- 聊天实时通信方式。
- 会话机制、token 策略和权限模型。
- 最低 Android 版本、目标 SDK 和发布渠道。

这些事项应在后续 `CONTEXT.md`、`docs/prd/` 和 `docs/adr/` 中逐步确认。
